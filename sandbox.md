本文档说明当前 Codex 会话可见的沙箱环境、能力边界、权限模型和工作原理。内容基于本会话暴露的运行配置编写；未公开的底层实现细节不会被假设为事实。

## 2. 沙箱的核心作用

沙箱的作用是给 Codex 提供一个受控执行环境，使其能够完成开发、调试、测试和文档编写任务，同时限制对系统其他区域的影响。

主要目标包括：

- 限制文件写入范围，避免误改用户系统或其他项目。
- 允许读取必要上下文，以理解当前项目和环境。
- 支持在项目目录内创建、修改、测试代码和文档。
- 对高风险操作引入人工批准。
- 将普通开发命令与越权命令区分开。
- 减少误执行破坏性命令的风险。
- 在可审计的上下文中执行任务。

## 3. 文件系统边界

当前文件系统权限使用受限配置。总体规则是：可以读取一部分上下文，可以写入 workspace 和临时目录，不能随意修改系统其他位置。

### 3.1 可写区域

以下路径允许写入：

- `/home/ningyuxie/MLWorkplaces/kimi_sandbox`
- `/tmp`
- 当前会话分配的临时目录

这意味着我可以在当前项目中：

- 新建文件，例如本文档 `sandbox.md`
- 修改已有项目文件
- 创建测试输出、缓存或构建产物
- 运行会在项目目录内写文件的工具
- 在 `/tmp` 中放置临时脚本、数据或中间产物

### 3.2 只读或受限区域

当前配置显示以下路径可读但不应写入：

- `/home/ningyuxie/MLWorkplaces/kimi_sandbox/.git`
- `/home/ningyuxie/MLWorkplaces/kimi_sandbox/.agents`
- `/home/ningyuxie/MLWorkplaces/kimi_sandbox/.codex`

根目录及其他系统路径通常也是只读或受限的。即使某些路径可以被读取，也不代表可以被写入。

### 3.3 写入沙箱外路径

如果任务需要写入 workspace 之外的位置，例如：

- `/usr`
- `/etc`
- `/var`
- 用户主目录下的其他项目
- 全局配置目录
- 系统服务目录

则通常需要显式授权。未经授权，我不应尝试绕过限制。

## 4. 命令执行边界

我可以运行 shell 命令来完成开发任务，例如：

- `ls`
- `pwd`
- `rg`
- `sed`
- `git status`
- `npm test`
- `pytest`
- 构建命令
- 格式化命令
- 本地开发服务器

命令默认在沙箱内执行。若命令涉及越权行为，则需要请求批准。

### 4.1 普通命令

普通命令通常可以直接运行，包括：

- 读取当前目录文件
- 搜索代码
- 查看日志
- 运行单元测试
- 运行项目本地构建
- 在 workspace 内生成文件

### 4.2 需要批准的命令

以下类型的命令通常需要用户批准：

- 写入沙箱外路径
- 修改系统目录
- 使用 `sudo`
- 启动 GUI 程序
- 打开外部应用
- 执行潜在破坏性操作
- 删除大量文件
- 修改全局环境
- 安装系统级依赖
- 访问受限硬件或服务

如果某个重要命令因为沙箱限制失败，并且完成任务确实需要它，我会重新以需要批准的方式请求执行。

### 4.3 破坏性操作

我不会擅自执行高风险破坏性命令，例如：

- `git reset --hard`
- `git checkout -- <file>`
- 删除用户未要求删除的文件
- 清空目录
- 重写历史
- 回滚用户已有改动

如果确实需要这类操作，必须先说明风险并获得用户明确同意。

## 5. 网络边界

当前会话网络访问已启用。网络可用于：

- 查询官方文档
- 下载依赖
- 访问包注册表
- 检索公开网页
- 调用允许的网络工具

但网络访问仍受任务需要和安全规则约束。

### 5.1 何时使用网络

当信息可能已经变化，或用户要求查询最新资料时，我需要联网确认。例如：

- 最新文档
- 当前版本号
- 价格
- 政策
- 法规
- API 变化
- 软件包状态
- 新闻
- 需要引用来源的问题

### 5.2 何时不使用网络

当任务可以完全基于本地上下文完成时，通常不需要联网。例如：

- 修改当前项目代码
- 总结用户提供的文本
- 编写本地文档
- 解释当前目录中的文件
- 运行本地测试

### 6.1 Shell 命令

通过命令行读取项目、运行测试、构建项目和启动服务。

典型用途：

典型用途：

- 新增文档
- 修改代码
- 调整配置

### 6.3 计划与进度

### 6.5 图像查看

## 7. 权限模型

当前权限模型可以理解为分层控制：

1. 默认在受限沙箱内执行。
2. workspace 和临时目录允许写入。
3. 沙箱外写入、系统修改和高风险操作需要批准。
4. 破坏性操作即使技术上可执行，也需要明确任务理由和用户同意。
5. 已有用户改动必须被保留，不能擅自回滚。

## 8. 沙箱工作原理

从可见配置推断，沙箱通过运行环境和工具层共同约束行为。可以将其理解为三层：

### 8.1 文件系统策略层

文件系统策略定义哪些路径可读、哪些路径可写、哪些路径受限。

当前策略要点：

- workspace 可写
- `/tmp` 可写
- 部分项目元目录只读
- 系统路径和其他项目路径默认受限

当命令试图写入不允许的位置时，可能失败，或需要提升权限。

### 8.2 命令执行策略层

命令默认以受限方式运行。系统会根据命令特征、路径和权限配置判断是否允许直接执行。

某些命令前缀可能已经被批准，可以在类似场景中复用；其他命令如果需要越权，则必须发起批准请求。

### 8.3 人工批准层

当操作超出默认权限边界时，系统要求我向用户提出批准请求。请求应包含：

- 要执行什么
- 为什么需要执行
- 可能影响什么范围

用户批准后，相关命令才可在更高权限下运行。

## 9. 沙箱不是完整虚拟机承诺

需要明确的是：从当前可见信息无法断言底层一定是完整虚拟机、容器、命名空间、seccomp、chroot 或某种特定内核隔离机制。

我能可靠说明的是：

- 当前会话呈现出受限文件系统权限。
- 命令执行受工具和权限策略约束。
- 写入范围被限制在 workspace 和临时目录。
- 越权操作需要用户批准。

不能可靠断言的是：

- 底层是否使用 Docker、Firecracker、gVisor、Bubblewrap 或其他技术。
- 进程隔离的具体实现。
- 网络隔离的底层实现。
- 内核级安全策略细节。
- 宿主机实际安全边界。

因此，本文档描述的是“当前 Codex 会话可见的执行沙箱”，而不是对底层平台安全架构的完整审计报告。

## 10. 对 Git 的处理原则

当前目录包含 `.git`，但该目录以只读方式暴露。

我可以执行常规 Git 查看类命令，例如：

- `git status`
- `git diff`
- `git log`
- `git show`

我不会擅自：

- 丢弃用户改动
- 强制切换分支
- 重置历史
- 删除未跟踪文件
- 改写提交历史

如果用户要求提交、变基或清理工作区，我会先检查状态，区分我做的改动和用户已有改动。

## 11. 对用户已有改动的保护

沙箱限制只是安全的一部分。行为规范同样重要。

我会遵守以下原则：

- 不覆盖用户未授权覆盖的文件。
- 不回滚用户已经做的修改。
- 在修改已有文件前先读取相关上下文。
- 若发现用户同时修改了相关文件，应基于现状继续工作。
- 若现有改动与任务冲突且无法安全合并，应向用户说明。

## 12. 临时文件与构建产物

可以在以下位置放置临时产物：

- 项目内合适的缓存或构建目录
- `/tmp`
- 会话临时目录

选择原则：

- 与项目有关且需要保留的产物放在项目中。
- 中间数据、一次性脚本和临时输出放在 `/tmp`。
- 不把大型无关产物写入项目。
- 不污染版本控制范围。

## 13. 本地服务

如果项目需要开发服务器，我可以启动本地服务并给出访问地址。

注意事项：

- 若默认端口被占用，应使用其他端口。
- 长时间运行的服务需要在结束前确认状态。
- 不应留下与任务无关的后台进程。
- 如果用户只需要静态 HTML，通常不必启动服务。

## 14. 安装依赖

依赖安装分为几类：

- 项目本地依赖：通常可以在 workspace 内执行。
- 用户级依赖：可能涉及主目录写入，需要视权限而定。
- 系统级依赖：通常需要批准。
- 全局工具安装：通常需要批准或至少需要明确理由。

如果依赖下载因网络或权限问题失败，我会根据失败原因决定是否请求提升权限。

## 15. 安全边界示例

### 15.1 允许的典型操作

- 创建 `sandbox.md`
- 修改项目源代码
- 新增测试文件
- 在项目目录运行测试
- 在 `/tmp` 生成临时数据
- 查询公开文档

### 15.2 需要谨慎或批准的操作

- 修改 `/etc/hosts`
- 安装系统包
- 写入另一个项目目录
- 删除大量文件
- 执行 `sudo`
- 启动 GUI 程序
- 改写 Git 历史

### 15.3 不应擅自做的操作

- 绕过沙箱限制
- 删除用户文件
- 回滚用户改动
- 将敏感文件上传到网络
- 在没有任务需要时扫描无关目录
- 运行与任务无关的高权限命令

## 16. 隐私与数据处理

在沙箱中工作并不意味着可以任意读取或传播数据。

应遵守以下原则：

- 只读取完成任务所需的文件。
- 不主动搜索无关敏感信息。
- 不把本地文件内容发送到外部服务，除非任务需要且用户知情。
- 对凭据、密钥、令牌等内容保持最小暴露。
- 如果发现敏感信息，避免在回复中复述完整内容。

## 17. 失败模式

常见失败模式包括：

- 权限不足导致写入失败。
- 命令试图访问受限路径。
- 依赖下载失败。
- 端口被占用。
- 测试需要系统服务但当前环境不可用。
- 项目依赖未安装。
- 命令需要 GUI 或硬件访问。

处理方式通常是：

1. 读取错误信息。
2. 判断是否为权限、网络、依赖或代码问题。
3. 优先在沙箱内解决。
4. 若确实需要越权，向用户请求批准。
5. 若无法解决，说明具体阻塞点和已验证内容。

## 18. 为什么沙箱仍然有用

即使底层实现细节不可见，当前沙箱仍提供实际价值：

- 限制默认写入范围。
- 将高风险操作变成显式批准流程。
- 降低误操作影响面。
- 让项目修改集中在 workspace 内。
- 支持可重复的开发和验证流程。
- 让用户清楚知道哪些操作会跨出默认边界。

- 可写目录
- 网络权限
- 已批准命令前缀
- 可用工具
- 运行 shell
- 当前日期和时区
- 是否允许启动服务
- 是否允许访问某些外部资源

如果需要确认新的会话环境，应重新检查环境上下文和权限配置。
